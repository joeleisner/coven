# Coven v0.2.0 Design Spec

Date: 2026-06-12

## Overview

v0.2.0 introduces three coordinated changes: a reorganized playground that showcases the full library, an expanded charm/hex API with clearly defined distinctions and a new export strategy, and a documentation pass to remove AI-style punctuation and reflect all new APIs.

---

## 1. Playground Reorganization

### File changes

- Delete `src/example.ts`, `src/example.css`
- Delete `src/_examples/` entirely
- Create `src/_playground/` with an entry point and one file per component
- Update `index.html` to load PicoCSS from a CDN link and point its module script at `src/_playground/index.ts`
- Remove `postcss-mixins` from `deno.json` imports (no longer building styles from scratch)
- Update `deno.json` `publish.exclude` to cover `src/_playground/**` instead of the old paths

No component-specific CSS files. PicoCSS handles base styles; components get minimal inline layout only if strictly necessary.

### Components

All three components follow progressive-enhancement first: meaningful light-DOM HTML works without JS. Shadow DOM and behavior are JS-only additions.

#### `coven-log`

Wraps an `<ol>`. With JS disabled, it renders as a plain ordered list.

With JS:
- Reads a `listen` attribute (space-separated event type names, e.g. `listen="counter-change toggle-visibility"`)
- Uses `$attr` to observe the `listen` attribute reactively
- Registers a document-level listener per event type via `$on`
- On each fired event, prepends a `<li>` to the `<ol>` with the event's `detail` as the displayed message
- Listeners are cleaned up via the element's signal

#### `coven-counter`

Wraps an `<output>0</output>` showing the current count. With JS disabled, it shows the static count value.

With JS:
- Shadow DOM adds `<button part="dec">` and `<button part="inc">` flanking a `<slot>`
- Uses `$attr` for the `count` attribute with a change callback that updates the slot display
- Dispatches a `counter-change` custom event via `$emit` on each change; `detail` carries the new value and direction (`'inc'` or `'dec'`)

#### `coven-toggle`

Wraps any slotted content. With JS disabled, the content is visible and unstyled.

With JS:
- Shadow DOM adds a `<button part="trigger">` above a `<slot>` that shows/hides the slotted content
- Uses `$prop` for internal `open` state
- Uses `$scry` (IntersectionObserver) to detect when the content enters or exits the viewport; dispatches a `toggle-visibility` event via `$emit` with a message in `detail`
- `coven-log` with `listen="counter-change toggle-visibility"` captures both components' events

## 2. Charm/Hex API

### Definitions

**Charm:** A bare, stateless helper. Takes an element as its first argument. Writes nothing to the grimoire. May accept an optional signal from the caller but does not manage signal lifecycle itself.

**Hex:** Adopts the bewitched paradigm. Always calls `$bewitch(element)` on entry (even if the signal is not directly used -- the contract guarantees the element is signal-ready after any hex call). Writes to the grimoire under a private symbol. Where a charm twin exists, the hex uses it for the core operation and layers grimoire state and signal management on top.

**Twin pairs:** Charm and hex twins share the same name (`$on`, `$shdw`, `$template`). They are distinct functions at different levels of abstraction. Pick one or the other; using both on the same element is only appropriate if you know exactly what each does to the grimoire.

### New charm twins

**`charms/$shdw`** -- Bare shadow attachment. Calls `attachShadow({ mode: 'open' })` and populates the root from an `HTMLTemplateElement` or HTML string. No grimoire writes, no parts tracking.

**`charms/$template`** -- Bare template creation. Creates and returns a fresh `HTMLTemplateElement` from an HTML string. No caching, no grimoire writes.

**`charms/$on`** -- Unchanged from current. Type-safe `addEventListener` wrapper. Accepts an optional signal from the caller.

### Updated hexes (now use charm twins)

**`hexes/$shdw`** -- Calls `$bewitch(element)`, then `charms.$shdw` for core attachment, then adds parts tracking and the internal `$mut` subscription. Exports its grimoire symbol tagged `@advanced`.

**`hexes/$template`** -- Calls `$bewitch(element)`, then `charms.$template` for core creation, then adds the per-class template cache via `grimoire.shared`. Exports its grimoire symbol tagged `@advanced`.

### New hex

**`hexes/$on`** -- Calls `$bewitch(element)` to ensure signal-readiness, then calls `charms.$on` with the element's signal threaded automatically. Callers no longer manage signal threading. Has no grimoire state of its own (relies entirely on `$bewitch`'s slot); no symbol is exported.

### New charm/hex pair (`$wake` / `$soul`)

These are not twins (different names, different scope). `$soul` uses `$wake` internally.

**`charms/$wake`** -- Bare deferred execution. Takes an element and a callback. Runs the callback immediately if `document.readyState !== 'loading'`, otherwise defers to `DOMContentLoaded`. Stateless; no signal, no grimoire writes.

**`hexes/$soul`** -- The functional alternative to extending `Familiar`. Accepts a `connected` callback and an optional `disconnected` callback. On call:
1. Calls `$bewitch(element)`, renewing the signal if already aborted (supports reconnection)
2. Uses `$wake` to run `connected(signal)` when DOM is ready
3. If `disconnected` is provided, registers it on the signal's `abort` event

For plain elements (not `Familiar` subclasses), the caller is responsible for triggering cleanup: call `$bewitch.abort(element)` when the element is removed from the DOM, which fires the `disconnected` callback.

Exports its grimoire symbol tagged `@advanced`.

### All other existing hexes

`$attr`, `$prop`, `$mut`, `$scry`, `$bewitch` already call `$bewitch` and write to the grimoire. No charm twins are introduced for these -- splitting off a stateless bare variant does not make sense given their designs. No changes to their internal behavior.

## 3. Export Strategy

### `deno.json` exports map

```json
{
  ".": "./src/mod.ts",
  "./charms": "./src/charms/mod.ts",
  "./hexes": "./src/hexes/mod.ts",
  "./familiar": "./src/familiar.ts",
  "./grimoire": "./src/grimoire.ts",
  "./hexes/$bewitch": "./src/hexes/$bewitch.ts",
  "./hexes/$attr": "./src/hexes/$attr.ts",
  "./hexes/$prop": "./src/hexes/$prop.ts",
  "./hexes/$mut": "./src/hexes/$mut.ts",
  "./hexes/$scry": "./src/hexes/$scry.ts",
  "./hexes/$shdw": "./src/hexes/$shdw.ts",
  "./hexes/$template": "./src/hexes/$template.ts",
  "./hexes/$on": "./src/hexes/$on.ts",
  "./hexes/$soul": "./src/hexes/$soul.ts"
}
```

### Main entry (`src/mod.ts`)

```ts
export { Familiar } from './familiar.ts';
export { grimoire } from './grimoire.ts';
export * as hexes from './hexes/mod.ts';
export * as charms from './charms/mod.ts';
```

`Familiar` and `grimoire` are flat exports (no naming conflict). `hexes` and `charms` are namespaced objects -- the import path declares the paradigm being adopted.

### Subpath entries

`src/charms/mod.ts` -- flat exports: `$assert`, `$define`, `$emit`, `$error`, `$on`, `$wake`, `$shdw`, `$template`

`src/hexes/mod.ts` -- flat exports of functions only (no symbols): `$bewitch`, `$attr`, `$prop`, `$mut`, `$scry`, `$on`, `$shdw`, `$template`, `$soul`

`src/familiar.ts` and `src/grimoire.ts` -- direct file exports; no changes to what they export.

Individual hex files (`src/hexes/$X.ts`) -- export the hex function as named + default, and their grimoire symbol tagged `@advanced`. Symbols are only reachable via the deep subpath import (`@joeleisner/coven/hexes/$shdw` etc.), never re-exported from `hexes/mod.ts`.

## 4. Documentation Updates

### Em-dash removal

All `—` (em-dash characters) across `docs/` and `README.md` are replaced with plain alternatives (commas, colons, or rephrased sentences). No other prose changes unless required by new API coverage.

### README

- "30-second example" headings become "Quick example"
- The `Familiar` quick example is updated to use the namespaced import style (`import { Familiar, hexes } from '@joeleisner/coven'`)
- A new "Quick example (plain element)" section showcases `$soul` with both callbacks and a cleanup comment:

```ts
import { hexes } from '@joeleisner/coven';

const el = document.querySelector('my-widget')!;

hexes.$soul(el, {
  connected: (signal) => {
    hexes.$on(el, { type: 'click', callback: () => console.log('clicked') });
  },
  disconnected: () => {
    console.log('cleaned up');
  },
});

// When removing from the DOM: hexes.$bewitch.abort(el)
```

- The "API at a glance" table is expanded with: `$wake` (charm), `$soul` (hex), `$shdw` (charm), `$template` (charm), `$on` (hex)
- A note is added distinguishing charm/hex twins (same name, pick one) from `$wake`/`$soul` (different names, different scope)

### Guide updates

- `docs/writing-a-charm.md` -- adds `$wake` and the new `$shdw`/`$template` charm skeletons; documents the twin-pair rule (pick one, not both)
- `docs/writing-a-hex.md` -- documents that all hexes call `$bewitch`; documents the twin pattern (use the charm for the core operation); adds `$soul` skeleton
- `docs/progressive-enhancement.md` -- adds `$soul` as the non-`Familiar` lifecycle path alongside the existing `$bewitch` pattern
- `docs/familiar-lifecycle.md` -- cross-references `$soul` as the functional equivalent to `Familiar`'s `connected`/`disconnected` hooks
- `docs/naming.md` -- adds `$wake` and `$soul` entries with etymologies
