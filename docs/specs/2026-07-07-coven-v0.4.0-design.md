# Coven v0.4.0 Design Spec

**Date:** 2026-07-07
**Branch:** v0.4.0

## Goals

1. Fix `$shdw` losing declarative shadow DOM content when it's parsed
   outside the streaming HTML parser (e.g. Astro's client-side router
   swap), which leaves a shadow host with an empty shadow root.
2. Audit every hex/charm's node parameter and widen it to the loosest
   DOM interface its implementation actually requires, instead of
   defaulting everything to `HTMLElement`.
3. Fix `$attr` never reflecting a default value onto the DOM and never
   firing its `callback` at setup.

---

## 1. `$shdw`: promote orphaned declarative shadow roots

**Problem:** `charms.$shdw` only checks `component.shadowRoot`:

```ts
if (!component.shadowRoot) {
  component.attachShadow({ mode: 'open' });
  if (html) { /* populate from html */ }
}
```

Declarative Shadow DOM (`<template shadowrootmode="open">`) is only
auto-promoted to a real `ShadowRoot` by the browser's streaming HTML
parser. Content built via `DOMParser.parseFromString(...)` (which is
how Astro's client-side router constructs the next page before
swapping it into the live document) does **not** promote declarative
shadow templates — they remain inert `<template>` elements sitting in
the light DOM.

Consequence: after an Astro view-transition navigation, a component
relying on DSD connects with `component.shadowRoot === null`. `$shdw`'s
guard treats this as "no shadow root yet," calls
`attachShadow({ mode: 'open' })`, and — since no `html` argument was
passed — leaves the new shadow root empty. The original declarative
markup is orphaned as an inert light-DOM `<template>`. Any query into
the shadow root (e.g. for a button) returns `null`.

**Fix:** before falling back to an empty `attachShadow`, look for a
declarative shadow template and promote it:

1. If `component.shadowRoot` already exists, behavior is unchanged
   (idempotent — return the existing root).
2. Otherwise, look for a direct-child `<template>` with a
   `shadowrootmode` attribute (`"open"` or `"closed"`).
   - If found: call `attachShadow` using that mode (and
     `shadowrootdelegatesfocus` / `shadowrootclonable` /
     `shadowrootserializable` attributes if present, mapped to their
     `ShadowRootInit` equivalents), move the template's `.content`
     into the newly attached root via `appendChild`, then remove the
     now-empty template from the light DOM.
   - If not found: fall back to today's behavior —
     `attachShadow({ mode: 'open' })`, then populate from `html` if
     provided.
3. An explicit `html` argument still takes precedence over promoting a
   leftover template — if the caller passes `html`, use it, matching
   current behavior. (In practice a component wouldn't pass both a
   real DSD template and an `html` string, but explicit input wins if
   it happens.)

This is the standard declarative-shadow-DOM polyfill promotion
pattern, applied lazily at `$shdw` call time rather than eagerly for
every element on the page — it only runs for elements that actually
call `$shdw`.

**Where:** `src/charms/$shdw.ts`. `src/hexes/$shdw.ts` calls the charm
internally, so it inherits the fix with no changes of its own.

**Testing:** `$shdw.test.ts` gains cases for:
- An element with a `<template shadowrootmode="open">` child and no
  `shadowRoot` yet → promotes the template's content into a real
  shadow root, removes the template.
- An element with `shadowrootmode="closed"` → promoted root has
  `mode: 'closed'`.
- An element with neither a real `shadowRoot` nor a template child →
  unchanged fallback behavior (empty `attachShadow`, or populated from
  `html` if given).
- An element that already has a real `shadowRoot` → unchanged
  idempotent behavior, no promotion logic runs.
- happy-dom support caveat: if happy-dom doesn't implement
  `shadowrootmode` template semantics, these tests construct the
  inert-template scenario manually (append a `<template
  shadowrootmode="open">` as a child without relying on the parser to
  promote it) — which is exactly the state Astro's `DOMParser` path
  produces, so it's a faithful reproduction either way.

## 2. Type-widening audit

**Principle:** type every hex/charm's node parameter to the loosest
DOM interface its implementation genuinely requires. Today almost
everything is typed `HTMLElement` regardless of what it actually
touches, which throws spurious TypeScript errors for `document`,
`window`, `SVGElement`, or other valid targets whose underlying DOM
calls work fine at runtime.

### Foundational widening: `grimoire` and `$bewitch`

`grimoire.ts`'s `GrimoireElement` type and `$bewitch`'s parameter
widen from `Node & GrimoireCarrier` to `EventTarget & GrimoireCarrier`.
Grimoire storage just stashes a value under a symbol key on the
object — it never depended on `Node`. This unlocks `window` (and any
other bare `EventTarget`: `AbortSignal`, `MediaQueryList`, etc.) for
every hex built purely on `$bewitch`.

### Per-function widening table

| Function | Current | New | Why |
|---|---|---|---|
| `charms.$on` | `HTMLElement` | `EventTarget` | only calls `addEventListener` |
| `$bewitch` (+ `.signal`, `.abort`, `.renew`) | `Node` | `EventTarget` | only stashes a signal in grimoire |
| `hexes.$on` | `HTMLElement` | `EventTarget` | `$bewitch` + `addEventListener` |
| `$soul` | `HTMLElement` | `EventTarget` | `$bewitch` + `$wake`; element param otherwise unused |
| `charms.$wake` | `HTMLElement` (unused) | `EventTarget` | unused; widened to match `$soul`'s new call convention |
| `$mut` (+ `.observer`, `.listeners`) | `Node` | `Node` (unchanged) | `MutationObserver.observe` requires a `Node` |
| `$attr` (+ `.list`) | `HTMLElement` | `Element` | `getAttribute`/`setAttribute`/`removeAttribute` + tagName (via `$assert`) |
| `$prop` (+ `.list`, `.readonly`) | `HTMLElement` | `Element` | tagName (via `$assert`); property definition itself is untyped-target-agnostic |
| `$scry` (+ `.observers`, `.disconnect`) | `HTMLElement` | `Element` | `IntersectionObserver.observe` requires `Element` |
| `$morph` (+ `.observers`, `.disconnect`) | `HTMLElement` | `Element` | `ResizeObserver.observe` requires `Element` |
| `charms.$shdw`, `hexes.$shdw` (+ `.parts`, `.root`, `.propagate`) | `HTMLElement` | `Element` | `attachShadow`/`shadowRoot`/tagName |
| `$template` (+ `.cache`, `.clone`) | `HTMLElement` | `Element` | tagName (via internal grimoire/class-cache path shared with `$shdw`) |
| `$emit` | `HTMLElement` | `Element` | tagName + `dispatchEvent` |
| `$assert` | `HTMLElement` | `Element` | tagName only (via `$error`) |
| `$error` | `HTMLElement` | `Element` | tagName only |
| `$define` | `typeof HTMLElement` | unchanged | Custom Elements spec requires an `HTMLElement` subclass |
| `Familiar` | extends `HTMLElement` | unchanged | same reason |

Every current call site already passes a real `HTMLElement`, which
satisfies every one of these widened types — this is purely additive
and does not break any existing usage.

**Where:** touches the parameter/return type annotations (and JSDoc
`@param` text where it references the old type name) in every file
listed above. No behavioral/runtime changes — this is a type-only
audit. `grimoire.ts` and `$bewitch.ts` are the only files with a
change to their actual generic constraint (`Node` → `EventTarget`);
everything else is a straight type-annotation swap
(`HTMLElement` → `Element` or → `EventTarget`).

**Testing:** existing tests are unaffected (they all use
`HTMLElement`s already). Add one or two smoke tests per widened tier
to lock the contract in:
- `$bewitch`/`hexes.$on`/`$soul` accept `window` without a cast.
- `$attr`/`$scry`/`$morph`/`$shdw`/`$emit` accept an `SVGElement`
  (e.g. `document.createElementNS('http://www.w3.org/2000/svg',
  'svg')`) without a cast.

**Docs:** update JSDoc examples/prose that currently say "any
`HTMLElement`" to reflect the new, more precise per-function
requirement (e.g. `progressive-enhancement.md` currently says "Coven
hexes work on any `HTMLElement`" — reword to "any `Element`" where
accurate, and call out `$on`/`$bewitch`/`$soul`'s further `EventTarget`
reach).

## 3. `$attr`: reflect defaults and fire the callback at setup

**Problem:** `$attr` (`src/hexes/$attr.ts`) delegates to `$prop` for
the property surface, and nests its own "reflect to attribute" and
"notify the caller's callback" logic entirely inside the callback it
hands to `$prop`. `$prop` (`src/hexes/$prop.ts`) only invokes its
callback when the property *setter* runs later — never at install
time (confirmed intentional there: `$prop.test.ts`'s `'$prop callback
fires on change'` test asserts exactly two calls after two external
assignments, none at setup). Since none of `$attr`'s own logic runs
until that callback first fires, two things silently never happen on
initial setup:

1. When the attribute is absent and `$attr` seeds the property from
   `value` (the default), the default is never written back to the
   DOM attribute — `element.hasAttribute(name)` stays `false` even
   though the property now holds the default.
2. The caller's `callback` never fires at setup at all, regardless of
   whether the attribute was absent or already present in markup.

(2) contradicts `$attr`'s own README usage example, which relies on
the callback firing once at setup to perform the initial render:

```ts
$attr<number>(this, {
  name: 'count', value: 0,
  callback: (v) => { $shdw.root(this)!.querySelector('span')!.textContent = String(v); },
});
```

As written today, the `<span>` never gets its initial "0" — only
later changes trigger it. This is a bug, not intended behavior; the
fix belongs in `$attr`, not `$prop` (whose no-fire-at-install contract
is already relied on and tested elsewhere as a raw property
primitive).

**Fix:** in `$attr`, immediately after `$prop(...)` returns
`initialValue`:

1. If the attribute was **absent** on the element
   (`element.getAttribute(name) === null`, captured before the `$prop`
   call), call the existing `reflectAttributeValue(initialValue)` once
   to seed the DOM. If the attribute was already present from markup,
   skip this — it's already in sync, and re-writing it would risk
   normalizing the author's original string formatting (e.g. `"007"`
   → `"7"`), which is out of scope for this fix.
2. If `callback` was provided and the signal isn't aborted, call it
   once with `callback.call(element, initialValue, initialValue)` —
   same value for old and new, since there's no real prior state to
   report. This runs unconditionally (regardless of whether step 1
   ran), matching the README's mount-time-render usage.

**Where:** `src/hexes/$attr.ts` only. No change to `$prop`, `$mut`, or
any other file.

**Testing:** `$attr.test.ts` gains cases for:
- Default value + attribute absent → `element.getAttribute(name)`
  reflects the default after the `$attr(...)` call returns (no
  external set needed).
- `callback` fires exactly once at setup with `(value, value)` when
  the attribute is absent, using the resolved default.
- `callback` fires exactly once at setup with `(value, value)` when
  the attribute is already present in markup, using the parsed
  attribute value — and the attribute string is untouched (no
  normalization) in that case.
- Existing `'$attr binds property to attribute'` test still passes
  unchanged (it only asserts behavior after an explicit property set,
  which remains correct).
